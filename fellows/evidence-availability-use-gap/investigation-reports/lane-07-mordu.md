## Part 1 — Why Mordu exists: components mapped to failure stages

Mordu's own doctrine (`MORDU-CHARTER-v0.md` §3, lines 34–46) names the load-bearing premise directly: *"knowing is not doing… a lesson sitting in context does not bind behavior; only external enforcement does."* Each component below is read against the audit's three stages — **Availability** (does the evidence exist and get surfaced), **Inspection** (does something actually look at it before a claim is made), **Binding** (does the result constrain what happens next, including across sessions).

## Part 2 — What Mordu does NOT address

1. **Runtime action authority beyond code-merge/deploy.** An AEVAE research thread lists exactly this gap in Mordu's own design, eleven days before the charter existed: *"The mordu-gate/GitHub approval design addresses whether Eve can merge code. It does not fully govern: sending emails, spending money, revealing data, contacting a person, accessing private machines, creating child agents, delegating authority, publishing content, changing memories, subscribing to services, manipulating physical systems."* (`memory/threads/2026/07/2026-07-10--t_iwokl73llwstjzci3yhrwq2mwt.md`, lines 376–388). The charter's own §6 jurisdiction table (lines 68–80) confirms this scope: destructive ops, secret-store, memory-deny, session budget, swarm fan-out, mirror staleness — all *system-operational* classes, none of them the action-authority classes the AEVAE thread names.

2. **Cross-surface capability availability** (whether a *third-party* connector — ChatGPT, Grok, a Claude Desktop connector — is actually callable in a given runtime, as opposed to merely configured). Nothing in `ledger.ts`, `sidecar-budget.ts`, or the charter's jurisdiction list touches this; it is a distinct failure class documented elsewhere in this audit's other lanes (their E8), and Mordu's server-side design has no visibility into a remote client's own tool registry.

3. **Correctness of a claim, only its shape.** `artifactProof()` (`src/ledger.ts` line 761) matches a *pattern* — a hex string 7–40 characters long, or a path prefix — not whether the referenced commit/file/test actually supports the claim being closed against it. A fabricated-looking-real commit hash would satisfy the regex. See MRD-07 below for a concrete instance where an inspection *ran* and still produced the wrong verdict because its check matched surface form, not substance.

4. **Duplicate/parallel work across sessions.** The fan-out gate (MRD-05, MRD-09) throttles *volume* of new agent spawns behind a go-word; nothing in the reviewed code detects that two or more independently-running sessions are reprocessing the same source material — which the threads show happening at least four times over roughly one day in early July (see MRD-09).

5. **Governance-surface growth itself.** Nothing in Mordu's design caps the number of new tickets, decisions, or proposed mechanisms that accumulate around Mordu's own construction (MRD-09's CHA-42/60/62/63/94 cluster). The "new-leg law" (charter §5.1, line 64) actually leans the other way — it is a rule for *adding* legs, with no corresponding rule for retiring or consolidating them.

6. **Its own documentation's accuracy.** Nothing enforces that `MORDU-STATE.md`/`MORDU-CHARTER-v0.md` prose stays synchronized with the code and config files sitting in the same repo — which is exactly what failed in MRD-02 and MRD-10 below.

## Part 3 — Mordu's own failure history

### MRD-01 — False-absence: "there is no Mordu repo" — **E3**

- **Claim:** MORDU-STATE.md's own context section reports that "a cloud session said *'there is no Mordu repo,'* which was wrong at the moment it was said, because it couldn't see the disk" (`MORDU-STATE.md` lines 59–61). `CODEX-P5-MORDU-CONTRACT.md` independently corroborates the same fact, more precisely: *"An older COS-session claim that 'no Mordu repo exists' came from a sandboxed session without repo visibility — it is wrong."* (line 21).
- **Availability:** The repo was real and on disk the whole time (`~/Documents/GitHub/mordu`, remote `yaw-sh/mordu`) — unavailable only to that one session's specific sandbox, not absent from the world.
- **Inspection:** The session asserted a global negative ("there is no X") from a local, scoped observation ("X is not visible to me") without qualifying the claim to its own visibility boundary.
- **Binding:** Downstream cost was real — this claim is the proximate reason `MORDU-STATE.md` had to be written at all, as a "single source of truth… verified from disk" corrective (`MORDU-STATE.md` line 3).
- **Note on evidence strength:** I could not locate the primary transcript for this claim inside the 15-file thread scope or a broader corpus grep; it is reported here exactly as two independent downstream documents describe it, not from a transcript I personally read. Flag accordingly — this is the one MRD item resting entirely on secondary sourcing.

  2. **AEVAE porting:** an AEVAE research thread, six days before the charter, already states the correct status table entry — *"Reusable COS/mordu components | **Exist elsewhere; not yet ported into EVE**"* (`memory/threads/2026/07/2026-07-10--t_iwokl73llwstjzci3yhrwq2mwt.md`, line 1672). - **Severity / recurrence:** High as a self-referential case — the artifact whose entire purpose is "receipts over promises" (clause II) was itself promise-without-receipt on its own founding facts.

### MRD-05 — Fan-out gate blocks a legitimate deep sweep pending a "go-word" — **E14 (contested)**

- **Source:** Same thread, lines 305–309 and 375–378.
- **What happened:** A four-lane "unknown-unknowns" sweep the session had designed could not run: *"Mordu's own fan-out gate — the guard added after a runaway multi-agent session burned most of a weekly quota — blocks any new fan-out unless your current message contains an explicit go-word."* The report frames this as *"the gate working as designed"* and asks the owner for one of three magic words ("ultracode," "fan out," "GATE:OK") to unblock it.
- **Reading against the charter:** The charter, written eleven days later, explicitly legislates against exactly this shape: clause V states *"Mordu checks receipts at action seams; it may not invent launch preconditions"* (line 58), and the acceptance test's second prong requires *"Refuses nothing during the oversight incident. No invented preconditions"* (§10, lines 99–100) — a direct response to a separate, earlier oversight-lockout incident the charter's doctrine table also cites (line 46: *"the oversight session invented an ENVIRONMENT CONTRACT precondition and every executor refused all work"*).
- **Assessment:** This is a genuinely contested case, not a clean incident, and I present it as such rather than force a verdict. In its favor: a go-word requirement is arguably a legitimate, narrow, class-scoped gate (fan-out volume after a real quota-burn incident), consistent with the "budget-ceiling" irreversible class the later contracts formalize (`src/contracts/shared.ts` lines 101–107). Against it: a magic-word unlock (rather than a structured "confirm this specific fan-out" prompt) is close in shape to an invented precondition, and the report itself treats the block as a real cost — it delayed a requested audit by a full session. I tag this **E14 (governance-induced-blocking)** with the explicit caveat that it may instead be correctly-functioning enumerated governance; the charter's own later anti-precondition clause suggests its authors saw the risk in this exact pattern even if they did not cite this incident by name.

### MRD-07 — Tautological recall-marker bug: Mordu's own check passed on a broken signal — **E5**

- **Source:** Same thread, lines 111–118.
- **What happened:** *"Mordu — the gate — checks at the end of every session whether the session actually consulted the brain. That check worked by searching the session transcript for a marker string that the recall hook prints. The bug: the hook printed a message containing that same marker string when recall failed. In other words, a session whose recall was completely broken still passed the 'did you use the brain' check."*
- **Why this is E5, not E1:** The gate did not skip inspection — it searched the transcript and found the marker, exactly as designed. The failure is that a present-marker was treated as proof of a successful recall when the marker's presence was in fact ambiguous between "recall succeeded" and "recall failed and said so." This is inspection running and still producing the wrong interpretation — the precise shape the task's E5 label names, and a useful caution for the audit's proposed "Verify" layer in general: a check that a receipt *exists* is not the same as a check that the receipt *means what it's assumed to mean*.
- **Binding:** Reported fixed the same night ("the failure message now announces itself as a failure in words the gate cannot mistake for success, and all sixty-two of the gate's tests plus the full hundred-and-eighty-one test suite pass," lines 116–118) — unverified independently by me (no test run performed in this audit; see Scope & method).

## Thesis verdict for Mordu

**Mordu strongly supports the thesis — and does so at two levels simultaneously, which is the most interesting finding of this lane.**

At the object level, Mordu's entire design is a direct, explicit response to the availability→inspection→binding gap: its founding doctrine table (charter §3) cites four independent incidents where the disqualifying evidence was already available and simply wasn't inspected or didn't bind — EVE-001 (MRD-08), the Keychain ban, the session-rot diagnosis, and BIG-MEMMA. Its core mechanisms (proof-required DONE, PIN-gated dispositions, session-close receipts, the fail-open/fail-closed split) are a fairly precise structural match to the audit's proposed Recall/Verify/Bind layer, built independently and for the same reason.

At the meta level — and this is where the evidence is strongest — **Mordu's own construction repeatedly reproduced the exact failure it exists to prevent.** MRD-02 (a ratified charter asserting a false state that the same corpus had already corrected six days earlier) and MRD-03 (a commit that claimed to "preserve all work" while deleting nearly half a million lines with zero insertions) are not incidents Mordu failed to catch in some *other* system — they are incidents in Mordu's own founding documents and Mordu's own deletion history. `MORDU-STATE.md`'s own words for MRD-03 — "There was no gate on the gate" — could stand as the thesis's title. MRD-10 extends this from documentation into infrastructure: a documented production-credential footgun was fixed in code before it was fixed in the branch that mattered, a gap only closed because an external audit went and checked, not because any mechanism inside Mordu itself would have caught an unmerged fix sitting in an orphaned worktree indefinitely.

The disconfirming material (MRD-15, and the resolved MRD-11/MRD-12) matters and should not be underweighted: where Mordu's claims were checked against a live external system (a real `wrangler deployments list` call, a real `npm test` run), they held up exactly as stated, including a claim more solidly confirmable than its own source documents dared assert. And the contested cases (MRD-05, MRD-13) show real design tension being actively worked, not ignored — the charter's own anti-precondition-invention clause reads like a direct response to the shape of incident MRD-05, even without citing it.

Net reading: Mordu is good evidence that the thesis's proposed fix is the right *shape* of intervention — deterministic, external, receipt-based, session-crossing — precisely because the same organization's informal alternative (more careful prose, more standing rules cited by number) visibly did not hold even inside the team building the formal fix. But Mordu is simultaneously proof that building the gate does not exempt the builders from the failure the gate targets — governance debt (MRD-09) and documentation drift (MRD-02) accrued *during* Mordu's own construction, at a pace roughly matching everything else in this corpus.
