# Lane 05 — GitHub Connector & Engineering-Plugin Case Study

**Thesis under test:** "Evidence availability–use gap" — evidence (tool/connector/credential state/file) was already available but the agent answered without inspecting it, shifting verification burden to the user.

**Scope:** `memory/threads/2026/07/` (grepped for connector/plugin/mcp.json/GitHub/Cowork/credential/token/authenticate/mounted/"not available"); `memory/familiars/claude/cowork/`; `memory/familiars/claude/local-agent-mode-sessions/` (Cowork "local agent mode" session tree under `44aa33d9-913a-4cfe-8be9-4df3f2bccaeb/c214d219-7cec-4bec-9cb8-a55ae9ff3918/`, ~48 session directories, `audit.jsonl` transcripts, `.claude.json` client state, `rpm/` plugin manifests). `memory/familiars/claude/cowork/KR/` and `/Mom/` were excluded as out-of-scope sensitive litigation/medical material unrelated to this lane.

**Method note:** All quotations below are either short technical strings (env var names, JSON keys, file paths, URLs) or tightly paraphrased. User frustration is described, not reproduced verbatim, per the privacy binding.

## 0. Verdict on the thesis

**Not falsified — substantially confirmed, with one important qualification.** The flagship incident (Section 1) shows a real "available-but-uninspected" gap: the GitHub connector's declared config (`.mcp.json` inside the Engineering plugin bundle) sat on disk, readable, across dozens of prior sessions, and was never opened until the user forced the question. Once forced, however, the agent's actual inspection was competent and honest — it read the real config, pulled real MCP logs, cross-checked GitHub's own docs, and clearly separated "declared" from "authenticated." So the pattern here is best described as **E2 (inspection gated behind user coercion) rather than E3 (fabricated false-absence claim) or E5 (inspected-but-misread)**. The qualification: I could not find a session where the agent claimed outright "GitHub isn't available" before being asked — the failure mode is closer to *never surfacing a known-broken capability proactively* than *lying about it when asked*. That is a real, still-damaging instance of the availability–use gap (the burden to notice and force verification fell entirely on the user), just a milder sub-type than the thesis's worst case.

A second, structurally identical incident (Section 2, a ChatGPT/OpenAI "Deep Research Review" thread, *not* Claude) shows the harsher version: an assistant that had not checked GitHub/Linear/MCP at all, said so only under direct, repeated challenge, and then conflated "not in my tool registry" with "not connected" — a genuine E3/E4 case. It is included because the user's archive treats it as part of the same operational corpus and because it sharpens the contrast with the Claude/Cowork case (Section 1's agent was more careful once triggered).

## 1. Flagship incident — Engineering plugin's GitHub MCP: declared, never authenticated

**Surface:** Claude Cowork (Claude Desktop "local agent mode"), model `claude-opus-4-8`.
**Path:** `memory/familiars/claude/local-agent-mode-sessions/44aa33d9-913a-4cfe-8be9-4df3f2bccaeb/c214d219-7cec-4bec-9cb8-a55ae9ff3918/local_2b782cb0-7748-4538-972d-7fa3c03530cc/audit.jsonl`, lines ~108–414 (415-line file, session runs 2026-06-26T00:30:58Z–01:12:56Z).
**Cross-referenced:** `.../rpm/manifest.json` (plugin registry), `.../rpm/plugin_01FTLa86dhbVJ3HB1LdHdhN7/.mcp.json` (Engineering plugin's declared MCP servers), and `mcp-needs-auth-cache.json` in ~48 sibling sessions.

**User objective:** Get help moving a crawler/email system ("mcp" → "cos" Cloudflare Worker migration) and, separately, get the GitHub connector working inside Cowork.

**What was available in-session, unused, before the trigger:** The session opened with the agent noting the "chamberlain memory MCP isn't connected" and asking a structured clarifying question (`AskUserQuestion`) about an unrelated topic ("What is COS?"). GitHub was not mentioned by the agent at all up to this point, despite the Engineering plugin (which bundles a `github` MCP server entry) having been installed and present in `pluginUsage`/`rpm/` since at least June 17 per `manifest.json` timestamps, and despite the same connector state (declared-but-dead) being consistent across essentially every session in the tree — `mcp-needs-auth-cache.json` was sampled in ~48 sessions spanning 2026-06 through 2026-07-16 and **`plugin:engineering:github` never once appears**, while `plugin:engineering:linear`, `:notion`, `:atlassian`, `:datadog` appear in nearly every one. This is the **available-but-uninspected** evidence: the connector's broken state was structurally invisible in the one place (`mcp-needs-auth-cache.json`) an agent would normally check quickly, because a connector whose OAuth handshake never completes doesn't land in that cache the way connectors with a working "needs-auth-click-to-fix" flow do. An agent skimming that cache file alone — without opening the plugin's own `.mcp.json` — would reasonably conclude "no known GitHub auth problem," which is false.

**Trigger:** The agent had posed a two-part structured question (COS meaning; how to proceed with 4 crawler files). The user answered off-script, in the free-text field, with a complaint that they had already looked in plugins and skills for GitHub, could not find/use it, called it unacceptable that it "should be in all of them." This is the coercion point: the agent's own next-step machinery (a menu of pre-set options) was hijacked by the user to force a topic the agent had not raised.

**Agent's initial response (line ~134):** "Hold acknowledged — I won't touch anything until the `mcp` repo is shared and memory's connected. But the GitHub thing — you're right to be annoyed, and it's *not* missing. Let me confirm exactly where it is." — Notably the agent does **not** claim false absence; it asserts presence *before* having looked (a plausibility-first move, E4), then immediately opens tools to verify: `ToolSearch("github repo pull request commit")`, `mcp__mcp-registry__search_mcp_registry`, then a targeted `ToolSearch("select:mcp__plugin_engineering_github__authenticate,...")`.

**Inspection (accurate, once performed):** At line ~195 the agent explains the actual architecture: GitHub ships bundled inside the Engineering plugin (along with linear/atlassian/notion/datadog/pagerduty), not as a standalone top-level entry — which is *why* a plugin/skill-name search for "GitHub" turns up nothing (a genuine discoverability gap, correctly diagnosed as such rather than denied). It then states plainly: "I just tried to load the GitHub connector's tools in this session and got nothing back... I won't pretend it's live when it isn't." This is the moment inspected-before-claim is satisfied — but only for the claim made *after* the user's redirect, not for the roughly 45 minutes of session time (and, per the cache evidence, weeks of prior sessions) before it.

At line ~302 the agent reads the actual declared config, `rpm/plugin_01FTLa86dhbVJ3HB1LdHdhN7/.mcp.json`:

— confirmed independently by direct file read in this investigation; the same file also declares `slack`, `linear`, `asana`, `atlassian`, `notion`, `pagerduty`, `datadog`, plus `"google calendar": {"url": ""}` and `"gmail": {"url": ""}` — **literal empty-string URLs**, i.e., genuine placeholder entries sitting next to GitHub's genuine (non-placeholder) URL in the same declared-config file. The agent then pulls real MCP logs (`Glob` for `*github*` under the Claude Code node cache), finds a `mcp-logs-plugin-engineering-github` directory exists (evidence of prior real connection attempts), is walled off from reading the app-internal cache dir, and instead runs a `WebSearch` to verify GitHub's actual remote-MCP auth requirements against official docs rather than guessing. Recommended fix: register a fine-grained Personal Access Token (PAT) via `claude mcp add --transport http github-pat ... --header "Authorization: Bearer ${GITHUB_PAT}"`, scoped to the relevant repos. The agent explicitly states it cannot enter the token itself ("credential rule") — **the user must do the secret-bearing step.**

**Did it bind?** Partially, and slowly:

- The underlying credential problem was **not resolved** in the observed window. A same-day follow-up self-audit (`local_4f603985-9f76-49ac-b4be-7737ed45ebc2/outputs/crawler-migration-merged-status.md`, verified against disk state 2026-06-26) contains an explicit claim-vs-verified table: *"GitHub MCP config fix — claimed: Patched to PAT — verified: **True**, `github-pat` config found in `~/.claude.json`"* directly next to *"GitHub PAT working — claimed: 'next step: create PAT' — verified: **Not done** — `GITHUB_PERSONAL_ACCESS_TOKEN` is **UNSET**, so the connector still can't auth."* Its own "switch map" lists `GitHub MCP (PAT): OFF — config done, token missing`.
- A status pull from a brain memory session four days later (`local_971636b4-e0b5-4433-8613-897e6bb98363`, referencing brain session `3a09c729`) still lists `GitHub connector PAT: ON, blocked on your PAT` — i.e., as of ~2026-06-29/30, still not done.
- Importantly, in that same session the agent **correctly separates connector auth from actual capability**: "I don't need the GitHub PAT to ship code — `gh` is logged in as `yaw-sh` with `repo` scope, so I can `git push` directly. The PAT only matters for the in-app connector." This is a good example of the agent *not* over-generalizing one broken credential surface into "GitHub access is unavailable" — a positive counter-example to E8 worth noting for balance.
- No session in the corpus shows the in-app GitHub MCP connector successfully returning tools after this point; the corpus is silent on final resolution.

**E-tags:** E1 (available-but-uninspected — the declared config and the cache anomaly were sitting unread across many prior sessions), E2 (inspection only happened after the user's off-script, forceful redirect), E4 (plausibility-first — "it's not missing" asserted before verification, though corrected within the same turn), E8 (cross-surface — in-app MCP OAuth vs `gh` CLI OAuth vs PAT-header MCP are three different credential surfaces, initially at risk of conflation, ultimately separated correctly), E15 (user-as-control-plane — user had to hijack an unrelated structured-question UI to force the topic onto the agenda).
**Best competing explanation:** the agent may not have "known" GitHub was even relevant until the user raised it, because nothing in the immediate task (a Cloudflare Worker migration) required GitHub tools yet — i.e., this could be read as reasonable task-scoping rather than negligence. This is a real mitigating factor and is why the severity below is rated moderate rather than severe.
**Severity:** Moderate. The connector had apparently been broken across many sessions/weeks without proactive surfacing, but once raised the agent diagnosed accurately and did not overclaim resolution.
**Smallest counterfactual prevention:** a session-start (or plugin-install-time) capability probe that actually calls each declared MCP server once and records real connect/fail state next to the declared config — so "declared" and "tool-count > 0" are reconciled automatically instead of only on user complaint. The existing `mcp-needs-auth-cache.json` mechanism is close to this but has a blind spot for connectors whose OAuth handshake never completes at all (they simply never populate the cache, in either direction).
**Confidence:** High. Directly read primary transcript with tool-call-level granularity; config claims independently re-verified against the actual `.mcp.json`/`manifest.json` files on disk.

## 6. Overall assessment

The GitHub-connector/engineering-plugin case study supports the evidence-availability–use-gap thesis, but the Claude/Cowork-specific evidence (Section 1, 3a–3d) is milder than the comparator ChatGPT case (Section 2): the recurring failure in this environment is **proactive silence about a known-broken capability across many sessions**, not **fabricating a false claim when asked**. Once the user forces the question, this environment's inspection is consistently accurate (real file reads, real log lookups, real external doc checks, and — notably — a same-day self-audit table that catches its own earlier overclaim about PAT completion). The structural root cause worth fixing is narrower and more tractable than "the agent doesn't check": it is that **one specific failure mode (an OAuth handshake that never resolves either way) is invisible to the one cache file (`mcp-needs-auth-cache.json`) an agent would normally consult**, so a quick self-check based on that cache alone would produce false confidence. A minimal Recall/Verify/Bind fix for this lane specifically: at plugin-install or session-start time, attempt one real tool-list call per declared MCP server and persist the actual result (connected / zero-tools-timeout / explicit-auth-needed) — not just whether an OAuth-needed flag was ever set — so "declared" and "functionally alive" are reconciled without requiring the user to notice and force it.
